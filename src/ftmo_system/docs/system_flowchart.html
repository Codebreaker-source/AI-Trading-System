Layer 2 · MQL5 Bridge EA · Runs every 3 seconds
BridgeEA_FTMO_v1.mq5 — Feature Writer & Trade Executor
The Bridge EA is the critical link between MetaTrader 5 and the Python system. It does two jobs simultaneously:

JOB 1 — Write Features (Python ← MT5)
Every 3 seconds it loops through all symbols in the Market Watch and calculates 27 CLEAN features per symbol, writing them to {SYMBOL}_features.csv in the MT5 Common\Files folder. Python reads these files to run predictions and strategies.

The 27 features are grouped as:
Price (4) close, high, low, volume Trend (8) SMA20/50, fast/slow EMA, HTF EMAs, HTF direction/alignment Momentum (4) RSI, Stoch K/D, Momentum Volatility (5) ATR, BB upper/mid/lower, volatility Volume (3) vol SMA, vol ratio, price×volume Sentiment (3) bullish/bearish/net sentiment

JOB 2 — Execute Trades (Python → MT5)
It scans Common\Files for any signal_{SYM}_{SOURCE}.txt files written by Python. For each signal found it parses the action (BUY/SELL), lot size, and trade ID, then validates and places the order. SL & TP are calculated by the EA from the live ATR — Python sends 0,0 to avoid wrong pip values on non-forex instruments like US500 or BTCUSD.

Position management: regime-adaptive trailing stops · partial TP at 2:1 R:R · break-even at 25% risk · progressive trail
Guard: SourcePositionExists() — one open position per (symbol, source) enforced at EA level
27-feature CSV files written to MT5 Common\Files per symbol

Layer 3 · Python Orchestrator · M15 Candle Cycle
live_trading_system.py — Feature Ingestion & Expansion
The main Python process wakes up on each M15 candle close and reads the latest feature CSV for every active symbol from the MT5 Common\Files path. This is a one-way data pull — Python never writes to MT5 directly, only through signal files.

Feature expansion: The raw 27 features are passed through feature_expander.py which derives 78 additional engineered features (ratios, crosses, normalised values, lagged indicators) producing a 105-feature vector — the exact format expected by the pretrained XGBoost models.

The raw 27-feature array is also kept as-is and passed directly to the strategy runner (EA translations and rule-based strategies don't need the full 105 features).

🔒 Layer 5 — 6-Phase Decision Gate  ·  Every signal must clear all 6 phases before an order is placed
P1
Capital Check — Account Safety
Before anything else, the system checks whether the account is still within safe drawdown bounds. It tracks equity in real-time against both FTMO's hard rules and the system's own conservative limits. If either limit is breached, all trading stops immediately — this is the most important safety feature on a prop firm account because breaching FTMO's limits means losing the funded account entirely.
PASS Equity > 97% of day-start balance AND > 90% total  ·  BLOCK Daily loss ≥ 3% (system) or ≥ 5% (FTMO hard) or total drawdown ≥ 10%
P2
Dimension Check — 4 Independent Assessments
Four separate dimensions evaluate the current market condition and cast votes (AGREE / DISAGREE / ABSTAIN).

🔵 REGIME dimension — Analyses the market regime (trending, ranging, volatile, low-vol). Blocks if regime is hostile to the signal direction (e.g. mean-reversion signal in a strong trend).

🟣 SESSION dimension — Only London (08:00–16:00 UTC) and New York (13:00–21:00 UTC) sessions are allowed. Trades during Tokyo or dead sessions are blocked — liquidity is too low.

🔵 ML dimension — XGBoost prediction for the 8 pretrained symbols. If the model disagrees with the signal direction it votes DISAGREE. If there is no model for this symbol, it returns ABSTAIN (not DISAGREE) — the symbol can still trade, it just needs stronger agreement from the other 3 dimensions.

🟠 CONFLUENCE dimension — An 8-factor weighted score (0.0–1.0) combining: candlestick patterns, support/resistance levels, HTF trend alignment, momentum, volume confirmation, BB position, RSI condition, and news sentiment. Must score ≥ 0.25 to pass.
PASS All 4 dimensions evaluated, votes recorded  ·  NOTE Symbols without ML: ML = ABSTAIN (3 active dimensions remain)
P3
Counter Vote — Minimum Agreement Required
The votes from Phase 2 are counted. The trade only proceeds if enough dimensions agree on the direction.

Symbols with ML model: Need ≥ 3 out of 4 dimensions to AGREE. One DISAGREE is allowed.

Symbols without ML model: The ML dimension is ABSTAIN, so 3 active dimensions remain. All 3 must AGREE — there's no room for disagreement when the ML safety net is absent.

No hard vetoes — no single dimension can unilaterally block a trade. The counter system means a strong majority wins, preventing one noisy signal from shutting down an otherwise solid setup.
PASS ≥ 3 dimensions AGREE (with model) · All 3 AGREE (without model)  ·  BLOCK Insufficient agreement
P4
Integration — Generate Final Signal
The outputs of all dimensions are combined into a single final signal with a direction (BUY/SELL) and confidence score. This is where the XGBoost confidence, confluence score, and dimension votes are weighted and merged.

The system also applies the HOLD-bias fix for XGBoost: if the model predicts HOLD but one of BUY or SELL has the second-highest probability, the strongest directional signal is used instead of blindly following the HOLD prediction — because models trained with imbalanced data tend to over-predict HOLD.
PASS Final direction confirmed with composite confidence score  ·  BLOCK Net confidence below threshold after integration
P5
Danger Scoring — 7-Category Risk Assessment (0–21 points)
An independent danger score is calculated across 7 categories. Each category can score 0–3 points. A total score ≥ 13 blocks the trade — too many risk factors active simultaneously.

SPREAD Is the current spread unusually wide (slippage risk)?
NEWS High-impact economic news event within 30 minutes?
STREAK Are we on a losing streak? (trade history tracker)
VOLATILITY Is volatility spiking beyond normal ATR range?
CORRELATION Are too many correlated pairs already open?
SESSION Are we in a low-liquidity transition period?
REGIME Is the regime unstable or rapidly changing?
PASS Danger score 0–12 · Manageable risk environment  ·  BLOCK Danger score ≥ 13 · Too many risk factors active
P6
Anti-Fragile Entry — Probe First, Build Later
Rather than entering with a full position immediately, the system uses an anti-fragile sizing approach. The first entry is always a probe — a minimal 0.01 lot order to test the water.

As the trade moves in the expected direction and reaches predefined R-multiple levels (e.g. 1:1, 1.5:1), the system adds to the position incrementally — building up to a maximum of 0.05 lots only when the market has confirmed the direction. Losing probes are small and cheap; winning trades get built up.

This means the system is never risking full size on an unproven signal. It earns the right to size up by being correct first.
PROBE 0.01 lot entry at signal price  ·  BUILD Add lots at R-levels as trade proves direction  ·  MAX 0.05 lot full position
Signal cleared all 6 phases → Python writes signal file

Layer 6 · File Bridge · MT5 Common\Files
Signal File Communication — Python → EA
Python and the EA communicate through flat text files in the MT5 Common\Files folder — a shared directory both processes can read and write. This is a deliberate design choice: it works regardless of how MT5 is launched, requires no sockets or network config, and is completely transparent (you can inspect any signal file in Notepad).

File name format: signal_{SYMBOL}_{SOURCE}.txt — e.g. signal_EURUSD.sim_dema_supertrend.txt
File contents (no header): symbol, action, confidence, sl_price, tp_price, lot, trade_id, timestamp

sl_price = 0, tp_price = 0 — Python intentionally sends zeros. The EA calculates live SL/TP from the current ATR, which correctly handles non-forex instruments (US500 point = 0.1, BTCUSD point = 0.01) without Python needing to know each symbol's pip value.

Python-side 15-minute cooldown per (symbol, source) — prevents Python from re-writing a signal file for the same source before enough time has passed for market conditions to change meaningfully.
EA reads signal files on next 3-second tick → validates and executes
💹
Layer 7 · Order Execution · EA-side validation
Trade Execution — EA Order Placement
The EA applies its own layer of validation before placing any order, acting as a final safety net against stale data, race conditions, or edge-case values from Python:

SourcePositionExists() — Checks if a position with comment "FTMO_AI_"+source is already open on this symbol. If yes, skip — one position per (symbol, source) maximum.
Minimum lot enforcement — Reads SYMBOL_VOLUME_MIN from MT5 per symbol (0.01 for forex, 0.1 for US500/US100, 0.01 for BTCUSD). Upgrades lot to minimum if needed.
ATR-based SL/TP calculation — BUY: sl = ask − 2×ATR, tp = ask + 3×ATR. SELL: sl = bid + 2×ATR, tp = bid − 3×ATR. Validated against SYMBOL_TRADE_STOPS_LEVEL.
Price normalisation — All prices normalised to the symbol's tick size before order submission to prevent "invalid stops" rejections.
Order comment — Set to "FTMO_AI_" + source + "_" + trade_id enabling full attribution in every position and deal record.
Fresh price fetch — ask/bid fetched immediately before PositionModify (not reused from earlier in the tick) to prevent race conditions on trailing stop modifications.
Order confirmed → EA and Python both log the execution
📋 EA Execution Log
File: execution_log/trades.csv

Written by the EA on every order fill. Contains: ticket number, symbol, action (BUY/SELL), lot size, entry price, SL, TP, trade_id, signal_source, and timestamp.

The trade_id links every EA execution back to the exact Python signal that triggered it, enabling cross-referencing between EA logs and Python logs.
📊 Unified Trade Logger
File: data/unified_trades.csv

Written by Python for every signal from every source — including signals that were blocked by the gate. Captures: trade_id, signal_source, source_type, action, confidence, confluence score, dimension votes, danger score, key indicators (close, RSI, ATR, momentum, volatility), whether the gate passed, whether execution was confirmed, and eventual outcome (TP/SL/manual, profit in pips and USD).

This is the primary dataset for later analysis — which sources work on which symbols, which gate phase blocks most trades, what conditions predict winners.
Daily at 00:00 UTC — retrainer reads execution log
🔄
Layer 9 · Daily Retraining · Sunday 00:00 UTC
daily_retrainer.py — XGBoost Auto-Creation from Execution Data
The retrainer is the system's self-improvement engine. As trades accumulate in the execution log, it periodically creates or updates XGBoost models — meaning symbols that start on rule-based-only automatically graduate to ML-enhanced trading once enough data exists.

Labelling logic: Each closed trade is labelled based on outcome — BUY hit TP → label BUY (correct) · BUY hit SL → label SELL (wrong direction) · SELL hit TP → label SELL (correct) · SELL hit SL → label BUY (wrong direction) · No trade taken → label HOLD. This teaches the model what market conditions actually predicted each direction.

Minimum 50 trades per symbol before training — below this threshold there's not enough data for a reliable model. The symbol stays on rule-based until the threshold is met.

Model replacement gate: The new model is only saved if its accuracy on a held-out test set matches or beats the previous model. A worse new model is discarded — the old one stays active. This prevents overfitting to a short bad run from replacing a better long-run model.

Output: data/models/{SYMBOL}_xgboost_CLEAN27.joblib — on next system startup, the symbol manager detects the new model and moves the symbol from the rule-based path to the full ML + confluence + 6-phase path.
