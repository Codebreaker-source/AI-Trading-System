📡 Layer 1 · Market Data Source — MetaTrader 5 (OANDA Terminal, OANDA-Demo-1)

The MT5 terminal supplies live tick/OHLCV for every symbol in Market Watch. No symbol list is hardcoded — the EA discovers whatever the broker exposes via SymbolsTotal(true), skipping disabled / "#" symbols.

Account note: running on OANDA demo 1600149577 ($200,000 balance, trade_allowed=True), attached EA on an M15 chart. config/ftmo_config.json uses simulation_mode: true and mt5.initialize() attaches to the already-running terminal (login 0 = no re-login). Despite the "FTMO" naming the system is on an OANDA demo today; FTMO credentials get filled in at the switch-over.
Every 3 s the EA timer fires → recompute indicators per symbol

🌉 Layer 2 · MQL5 Bridge EA · OnTimer every 3 s — BridgeEA_FTMO_v1.mq5

One EA on one chart drives both directions of the bridge through the MT5 Common\Files folder (FILE_COMMON), so it works no matter how MT5 is launched.

JOB 1 — Write features (MT5 → Python)

Each tick it loops all discovered symbols and writes {SYMBOL}_features.csv with 27 CLEAN features (overwritten every cycle):

Group#Features
Price4close, high, low, volume
Trend8SMA20/50, fast/slow EMA, HTF(H4) EMAs, HTF dir/align
Momentum4RSI, Stoch K/D, momentum
Volatility5ATR, BB up/mid/low, volatility
Volume3vol SMA, vol ratio, price×vol
Sentiment3bull/bear/net (derived from RSI)

JOB 2 — Execute (Python → MT5)

One FileFindFirst("signal_*.txt") pass per tick reads every signal file, splits signal_{SYM}_{SOURCE}.txt on the first underscore, and calls ExecuteTradeWithId() if action∈{BUY,SELL} and conf ≥ MinConfidence (0.45).

Also writes open_positions.csv (Python reads for position sync) and appends fills/closes to trades.csv. Re-discovers Market-Watch symbols every ~100 ticks (~5 min).

27-feature CSV per symbol in Common\Files

🐍 Layer 3 · Python Orchestrator · run_cycle() every 3 s — live_trading_system.py

Each cycle: (1) FTMO drawdown safety check — but in simulation_mode this returns immediately so signal generation (and training-data collection) never halts; (2) read the latest {SYMBOL}_features.csv for every symbol; (3) append the raw 27-vector to feature_history/ for sequence models; (4) update a 50-bar OHLC buffer for Fibonacci/pullback; (5) expand 27→105 via feature_expander.py.

Reality check: the XGBoost models need exactly 27 inputs, and make_ml_predictions() uses features[:27], so the 105-feature expansion is logged but not consumed by the ML path. Rule strategies pull a few named features from the expanded dict (and several — currency strength, returns_std — aren't in the 27, so they default to neutral).
features uploaded to Azure (if enabled) · 27-vector fed to all local sources

☁️ Layer 3b · Parallel Cloud Inference · azure_bridge.py

Azure Blob ↔ Google Colab Ensemble (LightGBM · CatBoost · XGBoost · Transformer)

The cloud models run on Colab's GPU, not locally. The local process uploads each symbol's feature CSV to the Azure container trading-features; a Google Colab runtime (colab/trading_inference.ipynb) reads them, runs LightGBM + CatBoost + XGBoost + Transformer inference, and writes prediction JSONs back to colab-predictions as {SYMBOL}_{source}_pred.json (~every 30 s). v3: XGBoost was added to the cloud ensemble — Colab now clones a full 49-symbol set of all four model types from the repo.

Local side polls azure.get_all_predictions() (now over sources lgbm, catboost, xgboost, transformer); predictions older than 1800 s (30 min) are discarded as stale, and a warning logs if no fresh predictions arrive for 10 min (Colab runtime likely disconnected). Auth is the AZURE_STORAGE_CONNECTION_STRING env var — if unset, AzureBridge.enabled = False and this entire path is silently skipped (local sources still run). Colab access itself is a browser session in colab/edge_profile/ kept alive by colab/keepalive.py.

4 cloud predictions join the local sources at Layer 4 — as UNGATED sources (Path B)

🔎 Path A pre-filter · apply_confluence_filtering()

Hard Filters → Regime → Confluence Score → Portfolio Risk

Hard filters (min_atr_pips=8, 30-min news buffer, liquid-session). SIM require_liquid_session is set to False in simulation_mode, so Asian/off-hours pass here to maximise labelled data.

Regime detector — ADX-based: trending ≥25, ranging ≤20, else volatile.

Confluence scorer — weighted blend: mtf_trend 0.25 · support_resistance 0.20 · strategy_consensus 0.20 · momentum 0.15 · volume 0.10 · volatility 0.10. Must score ≥ 0.35 (config confluence_threshold) to proceed. CORRECTED baseline said "8-factor ≥ 0.25".

Portfolio risk — RiskManager blocks new entries if portfolio risk ≥ 2% of trading capital (10% of account). v3 fix: risk is now priced from mt5.symbol_info tick_size/tick_value (forex table fallback) instead of a hardcoded forex pip table — the old default mispriced indices/metals/crypto so badly it pinned portfolio risk at 208.5% and blocked every signal.

survivors enter the 6-phase decision gate

🔒 Layer 5 — 6-Phase Decision Gate · PATH A (XGBoost) ONLY · inside generate_signals()

P1Capital Check — FTMO / system drawdown

Run at the top of run_cycle(). Daily loss ≥ 3% (system) / 5% (FTMO) or total DD ≥ 10% halts all trading. SIM bypassed entirely in simulation_mode (check_ftmo_drawdown_safety() returns False) — so in the current sim run this phase never blocks. FLAG the EA enforces its own copy of these limits regardless (Layer 6).

PASS within limits / sim mode · BLOCK live mode + limit breached
P2Dimension Check — 4 dimensions (AGREE / ABSTAIN / DISAGREE)
  • 🔵 REGIME — TRENDING→AGREE; RANGING/VOLATILE→ABSTAIN; conf<0.4→ABSTAIN.
  • 🟣 SESSION — London 07:00–16:00 & NY 12:00–21:00 UTC→AGREE; overlap 12:00–16:00 best; JPY-Tokyo→ABSTAIN; hour<7 or ≥21→DISAGREE (veto). CORRECTED hours vs baseline's 08:00/13:00.
  • 🔵 ML — XGBoost; matches dir + agreement≥0.67→AGREE; opposite→DISAGREE; no model / HOLD / low-conf→ABSTAIN.
  • 🟠 CONFLUENCE — ≥0.35→AGREE; 0.25–0.35→ABSTAIN; <0.25→DISAGREE (veto).
VETO any single DISAGREE blocks the trade — FLAG baseline wrongly said "no hard vetoes"
P3Counter Vote — ≥ 3 / 4 AGREE and no veto

can_trade = (agrees ≥ 3) AND (no DISAGREE). With a model: 3/4 needed (one ABSTAIN tolerated). Without a model: ML abstains, so all 3 remaining must AGREE. (P2/P3 are one computation in DimensionChecker.check_all(); split here for readability.)

PASS ≥3 AGREE, 0 DISAGREE · BLOCK otherwise
P4Integration — direction, SL/TP, scale-in validation

Skip HOLD; require confidence ≥ 0.35; optional news confidence adjustment. SL/TP = 1.5× / 3.0× ATR (fixed 2:1). CORRECTED baseline said 2×ATR. Scale-in only if: same direction, < 3 positions, at breakeven, ≥35% of reward reached, confluence ≥ 0.35, and pullback valid; scale-in SL = 75% of prior risk.

PASS direction + levels set · BLOCK HOLD / conf<0.35 / scale-in invalid
P5Danger Scoring — 7 categories, 0–21, ≥ 13 blocks

DangerScorer sums points across regime · session · ml · technical · stress (drawdown/losing streak) · correlation (same-direction heat) · event (mins-to-high-impact news). Score ≥ 13 blocks; otherwise yields a size_multiplier (currently logged, not yet applied to lot).

PASS 0–12 · BLOCK ≥ 13
P6Entry — flat 0.01 (anti-fragile builder DISABLED in v3)

v3: FLAT_LOT=0.01 + DISABLE_SCALING=True. Every fresh signal opens exactly one 0.01 position — no probe→build, no scale-in adds, and check_scaling_opportunities() returns early (no partial scale-outs; you can't partial-close a 0.01 position). If a symbol already has an open position, the new signal is skipped (one position per symbol). The old probe 0.05 → build 0.15 anti-fragile path is bypassed. Rationale: a clean 1:1 signal↔PnL mapping to verify sim PnL against the MT5 broker files and gather uniform data.

FLAT 0.01 per signal · no build · no scale-in · no scale-out
Path A survivor → write_trade_commands() → per-symbol signal file

📄 Layer 6 · File Bridge · MT5 Common\Files — Signal Files (both paths converge)

Both Path A and Path B emit the same artifact: signal_{SYMBOL}_{SOURCE}.txt, one line, no header: symbol, action, confidence, sl_price, tp_price, lot, trade_id, timestamp.

sl_price = tp_price = 0 by design — Python lets the EA compute SL/TP from live ATR so non-forex instruments (US500 point 0.1, BTCUSD) get correct pip maths. The trade_id carries attribution end-to-end. Python keeps a 15-min cooldown per (symbol, source) before re-writing.

EA picks up the file on its next 3-s tick

💹 Layer 7 · Order Execution · ExecuteTradeWithId() — EA-Side Validation & Placement

The executed code path checks, in order:

  • EnableTrading and CheckFTMOLimits() — daily 3%/5%, total 10% halts.
  • SourcePositionExists() — if a position commented FTMO_AI_{source} is already open on the symbol, skip. One position per (symbol, source).
  • MinConfidence 0.45 gate at read time — FLAG Python's Path-A threshold is 0.35, so XGBoost signals with 0.35–0.45 confidence pass Python but are rejected here. Strategy signals send conf 1.0, so they always clear.
  • Min-lot + lot-step normalisation per symbol (0.01 forex, 0.1 US500…).
  • Spread ≤ MaxSpreadPips (5); else skip.
  • ATR SL/TP: BUY sl = ask − 1.5·ATR, tp = ask + 3.0·ATR (and inverse for SELL), each validated against SYMBOL_TRADE_STOPS_LEVEL and normalised to digits.
Dormant guards: portfolio-risk %, correlation-group cap (max 2), max-total-positions (10) and streak-based sizing live in the other function ExecuteTrade(), which the signal scanner never calls. On the live signal path they are not enforced — only FTMO DD, spread, one-per-source and min-lot are.
order opened → managed continuously, then logged on close

🎚️ Layer 7b · ManagePositions() · every 5 s, all magic-number positions

Break-Even · Regime Trailing · Partial TP · Progressive Trail

Applies to every open position regardless of which path opened it: Break-even at 0.25R (+2-pip buffer); Partial TP closes 50% at 2:1 then extends TP to 3:1 and moves SL to BE; regime-adaptive trailing (ATR× 1.5 ranging / 2.5 trending / 3.5 volatile) tightened by progressive tiers (×0.9 / 0.75 / 0.5 at 1.0 / 1.5 / 2.0 R). Fresh bid/ask fetched immediately before each modify to avoid race conditions.

on close → outcome labelled and logged (both sides)

📋 Logging — two sinks

EA Execution Log — trades.csv (Common\Files)

On every fill/close: ticket, symbol, direction, 3-tier outcome — TP (take-profit hit), TRAIL (SL reason but profit>0), SL (loss) — entry/exit, volume, profit, trade_id, signal_source (parsed from the FTMO_AI_{source} comment). Drives streak tracking and the retrainer.

Unified Trade Logger — data/unified_trades.csv (Python)

One row per signal from every source (gated or not): trade_id, source, source_type, action, confidence, confluence, dimension votes/count, danger score, key indicators (close, RSI, ATR, momentum, volatility), would_execute / actually_executed. Outcomes (TP/SL/PnL/MFE/MAE) are filled by the background trade_outcome_simulator.py (tick replay). v3 fix: its profit_usd now uses real per-symbol contract economics (was a hardcoded forex 100k contract → non-forex PnL overstated up to 100,000×). Pre-2026-06-22 rows are corrupted and should not be used for training/perf decisions.

Sunday 00:00 UTC — retrainer consumes the execution log

🔄 Layer 9 · Self-Improvement · daily_retrainer.py — DISABLED in v3

Per-source retrain from own-trade outcomes (Option A)

Currently OFF — retraining.enabled=false and run_system.py skips the thread. Shelved until two things land: (1) the retrainer is reworked to Option A, and (2) enough clean post-fix own-trade data accumulates.

Option A rule (hard requirement): each ML model trains ONLY on the outcomes of trades it itself generated — filter unified_trades.csv by signal_source, label each by how price resolved it (TP→correct dir, SL→opposite, via data_labeler.label_trade). A model is NEVER trained on another source's trades, and NOT on a shared full-history price-action label set. The current daily_retrainer.py still does the latter (Option B) and must be reworked before re-enabling. Rule-based strategies are never retrained.

🌱 Seeds vs. retrain: all 49 symbols are seeded from ~3yr Dukascopy price-action labels (forward 20-pip/24-candle) via the download→extract→label→pretrain pipeline — that's the bootstrap. Option-A own-trade retraining is the live improvement loop layered on top once enabled.
